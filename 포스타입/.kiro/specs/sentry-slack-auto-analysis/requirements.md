# 요구사항 문서

## 소개

Sentry에서 Slack으로 오류 알림이 이미 전달되고 있는 환경에서, 해당 알림 메시지의 스레드에 자동으로 오류 원인 분석과 해결 대안을 제공하는 자동화 시스템이다. 장애 상황에서 개발자가 코드나 인프라를 직접 확인하지 않더라도 빠르게 원인을 파악하고 대응할 수 있도록 지원하는 것이 핵심 목적이다.

## 용어 정의

- **워크플로우_시스템**: Slack에 도착한 Sentry 오류 알림을 감지하고, LLM 기반 분석 결과를 해당 메시지의 스레드에 자동 응답하는 자동화 시스템
- **Slack_봇**: Slack API를 통해 채널의 메시지를 감지하고 스레드에 답변을 작성하는 봇 애플리케이션
- **오류_이벤트**: Sentry에서 감지되어 Slack으로 전달된 개별 오류 알림 메시지
- **분석_결과**: 오류 추정 원인, 영향 범위, 해결 대안, 담당자 정보를 포함한 구조화된 응답
- **담당자_매핑**: 오류 유형 또는 프로젝트별로 담당 개발자를 매핑한 설정 정보
- **LLM_분석기**: 오류 정보를 입력받아 원인 분석과 해결 대안을 생성하는 대규모 언어 모델 기반 분석 모듈
- **Sentry_MCP**: Sentry MCP(Model Context Protocol) 서버를 통해 이슈 상세 정보 조회, 이벤트 검색, AI 기반 근본 원인 분석 등을 수행하는 도구 인터페이스. 주요 도구: `get_issue_details`, `search_issues`, `search_events`, `analyze_issue_with_seer`

## 요구사항

### 요구사항 1: Sentry 알림 메시지 감지

**사용자 스토리:** 개발자로서, Slack에 도착한 Sentry 오류 알림을 시스템이 자동으로 감지하길 원한다. 이를 통해 수동 개입 없이 분석 워크플로우가 시작된다.

#### 인수 조건

1. WHEN Slack 채널에 새 메시지가 도착하면, THE 워크플로우_시스템 SHALL 해당 메시지가 Sentry 알림인지 판별한다
2. THE 워크플로우_시스템 SHALL Sentry 봇 사용자 ID 또는 메시지 내 Sentry 링크 패턴을 기준으로 Sentry 알림을 식별한다
3. IF 메시지가 Sentry 알림이 아니면, THEN THE 워크플로우_시스템 SHALL 해당 메시지를 무시하고 추가 처리를 수행하지 않는다
4. WHEN Sentry 알림이 감지되면, THE 워크플로우_시스템 SHALL 해당 메시지의 타임스탬프(thread_ts)를 기록하여 스레드 응답에 사용한다

### 요구사항 2: Sentry 오류 상세 정보 수집

**사용자 스토리:** 개발자로서, Sentry 알림에 포함된 오류의 상세 정보가 자동으로 수집되길 원한다. 이를 통해 코드를 직접 보지 않아도 충분한 컨텍스트를 확보할 수 있다.

#### 인수 조건

1. WHEN Sentry 알림 메시지가 감지되면, THE 워크플로우_시스템 SHALL 메시지에서 Sentry 이슈 URL 또는 이슈 Short ID를 추출한다
2. WHEN Sentry 이슈 Short ID가 추출되면, THE 워크플로우_시스템 SHALL Sentry_MCP의 `get_issue_details` 도구를 호출하여 오류 메시지, 스택트레이스, 프로젝트명, 오류 레벨, 발생 환경(environment), 태그, 브레드크럼(breadcrumbs) 정보를 조회한다
3. IF Sentry_MCP 도구 호출에 실패하면, THEN THE 워크플로우_시스템 SHALL Slack 메시지 본문에서 추출 가능한 정보만으로 분석을 진행한다
4. THE 워크플로우_시스템 SHALL 수집된 오류 정보를 구조화된 형식으로 변환한다
5. WHEN 오류 상세 정보가 수집되면, THE 워크플로우_시스템 SHALL Sentry_MCP의 `analyze_issue_with_seer` 도구를 호출하여 Sentry Seer AI 기반 근본 원인 분석 및 코드 수정 제안을 조회한다

### 요구사항 3: LLM 기반 오류 원인 분석

**사용자 스토리:** 개발자로서, 장애 상황에서 코드나 인프라를 직접 확인하지 않아도 오류의 추정 원인과 대응 방안을 즉시 파악하길 원한다. 이를 통해 초기 대응 시간을 단축할 수 있다.

#### 인수 조건

1. WHEN 오류 상세 정보가 수집되면, THE LLM_분석기 SHALL 오류 메시지, 스택트레이스, 브레드크럼 정보를 기반으로 추정 원인을 분석한다
2. THE LLM_분석기 SHALL 분석 결과에 추정 원인 요약, 영향 범위, 관련 코드 영역 추정, 발생 가능한 시나리오를 포함한다
3. THE LLM_분석기 SHALL 분석 결과에 최소 1개 이상의 구체적인 해결 대안 또는 즉시 대응 가능한 조치를 제시한다
4. IF LLM API 호출에 실패하면, THEN THE 워크플로우_시스템 SHALL 오류 정보 요약만 포함한 기본 응답을 생성한다
5. THE LLM_분석기 SHALL 분석 요청당 응답을 30초 이내에 생성한다

### 요구사항 4: 담당자 자동 매핑

**사용자 스토리:** 개발자로서, 오류 발생 시 적절한 담당자가 자동으로 지정되길 원한다. 이를 통해 책임 소재가 명확해지고 대응 속도가 빨라진다.

#### 인수 조건

1. WHEN 오류_이벤트의 프로젝트명과 오류 유형이 확인되면, THE 워크플로우_시스템 SHALL 담당자_매핑 설정에서 해당하는 담당자의 Slack 사용자 ID를 조회한다
2. IF 담당자_매핑에 일치하는 담당자가 없으면, THEN THE 워크플로우_시스템 SHALL 기본 담당자(default assignee)를 지정한다
3. THE 워크플로우_시스템 SHALL 담당자_매핑 설정을 JSON 또는 YAML 형식의 설정 파일로 관리한다

### 요구사항 5: Slack 스레드 자동 응답

**사용자 스토리:** 개발자로서, Sentry 오류 알림의 스레드에 분석 결과가 자동으로 달리길 원한다. 이를 통해 별도의 도구 전환 없이 Slack에서 바로 원인과 대응 방안을 확인할 수 있다.

#### 인수 조건

1. WHEN 분석_결과가 생성되면, THE Slack_봇 SHALL Sentry 알림 메시지의 스레드에 답변으로 분석 결과를 게시한다
2. THE Slack_봇 SHALL 스레드 응답에 오류 요약, 추정 원인, 영향 범위, 해결 대안, 즉시 대응 조치, 담당자 멘션(@사용자), Sentry 이벤트 링크를 포함한다
3. THE Slack_봇 SHALL 스레드 응답을 Slack Block Kit 형식으로 구조화하여 가독성을 확보한다
4. IF Slack API 호출에 실패하면, THEN THE 워크플로우_시스템 SHALL 실패 내용을 로깅하고 최대 3회까지 재시도한다
5. IF 3회 재시도 후에도 실패하면, THEN THE 워크플로우_시스템 SHALL 관리자에게 별도 알림을 전송한다

### 요구사항 6: 중복 처리 방지

**사용자 스토리:** 개발자로서, 동일한 오류에 대해 중복 분석이 수행되지 않길 원한다. 이를 통해 불필요한 알림 노이즈를 줄일 수 있다.

#### 인수 조건

1. WHEN 동일한 Sentry 이슈 ID에 대한 알림이 설정된 시간 내에 중복 감지되면, THE 워크플로우_시스템 SHALL 중복 요청을 무시한다
2. THE 워크플로우_시스템 SHALL 중복 판별을 위해 최근 처리된 이슈 ID를 캐시에 보관한다
3. THE 워크플로우_시스템 SHALL 중복 판별 시간 창(time window)을 설정 가능하게 한다

### 요구사항 7: 설정 관리

**사용자 스토리:** 관리자로서, 워크플로우의 동작을 설정 파일로 관리하길 원한다. 이를 통해 코드 변경 없이 동작을 조정할 수 있다.

#### 인수 조건

1. THE 워크플로우_시스템 SHALL 감시 대상 Slack 채널 ID, LLM API 키, 담당자_매핑 정보를 설정 파일에서 읽어온다 (Sentry 인증은 MCP 서버가 처리)
2. WHEN 설정 파일이 변경되면, THE 워크플로우_시스템 SHALL 재시작 없이 변경된 설정을 반영한다
3. IF 필수 설정 값이 누락되면, THEN THE 워크플로우_시스템 SHALL 시작 시 명확한 오류 메시지와 함께 종료한다
4. THE 워크플로우_시스템 SHALL API 키와 토큰을 환경 변수 또는 시크릿 관리 도구를 통해 관리한다
