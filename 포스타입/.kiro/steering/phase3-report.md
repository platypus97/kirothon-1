# Phase 3: RCA 보고서 생성 및 슬랙 게시

이 단계에서는 Phase 1(Sentry 컨텍스트)과 Phase 2(코드 분석)의 결과를 종합하여 RCA(Root Cause Analysis) 보고서를 생성하고, 사용자 확인 후 원본 슬랙 스레드에 게시한다.

---

## 1. RCA 보고서 생성

Phase 1에서 수집한 Sentry 에러 정보와 Phase 2에서 분석한 코드 분석 결과를 종합하여 RCA 보고서를 작성한다.

### 1-1. 보고서 구성 섹션

RCA 보고서는 다음 6개 섹션으로 구성한다. 모든 섹션을 빠짐없이 포함해야 한다.

**지시사항:**

아래 템플릿에 따라 RCA 보고서를 작성한다. 각 섹션의 내용은 Phase 1, 2의 분석 결과에서 가져온다.

**보고서 템플릿:**

```
🔍 RCA 보고서 — Root Cause Analysis

========================================
📋 1. 이슈 요약
========================================

- 에러 타입: <Sentry 이슈 title>
- 발생 위치: <culprit 또는 스택트레이스 최상단 프레임의 파일:라인>
- 발생 횟수: <count>회
- 영향 사용자: <userCount>명
- 최초 발생: <firstSeen>
- 최근 발생: <lastSeen>
- Sentry 링크: <issueUrl>

(여러 이슈가 있는 경우 이슈마다 위 블록을 반복)

========================================
🔥 2. 에러 상세
========================================

**스택트레이스 (inApp 프레임):**
1. <filename>:<lineNo> — <function>
2. <filename>:<lineNo> — <function>
...

**최근 이벤트 컨텍스트:**
- 브라우저: <browser>
- OS: <os>
- 환경: <environment>
- 릴리즈: <release>
- 요청 URL: <request url> (있는 경우)

**태그 정보:**
- <key>: <value>
...

========================================
🔎 3. 근본 원인 분석
========================================

**직접 원인:**
<에러를 직접 발생시킨 코드와 조건에 대한 설명>

**근본 원인:**
<직접 원인이 발생하게 된 상위 원인에 대한 설명>
(확실하지 않은 부분은 [추정] 태그를 붙인다)

**확신도:** <높음 / 중간 / 낮음>

**호출 체인:**
<에러 발생 지점에서 상위 호출자까지의 호출 흐름>

**관련 코드 경로:**
1. <파일 경로> (line <lineNo>) — <해당 코드의 역할>
2. <파일 경로> (line <lineNo>) — <해당 코드의 역할>
...

**추정 근거:**
<해당 원인으로 추정한 이유와 분석 과정에서 발견한 근거>

========================================
🛠️ 4. 즉시 조치 방안
========================================

**조치 #1: <조치 제목>**
- 설명: <조치에 대한 설명>
- 코드 수정 제안:
  <구체적인 코드 변경 내용 또는 수정 방향>
- 대상 파일: <파일 경로>

(조치가 여러 개인 경우 반복)

========================================
🛡️ 5. 재발 방지 제안
========================================

1. <재발 방지를 위한 제안 사항>
2. <재발 방지를 위한 제안 사항>
...

========================================
📝 6. 분석 노트
========================================

**건너뛴 단계:**
(건너뛴 단계가 있는 경우에만 표시)
- <단계명>: <건너뛴 사유>

**파일 미발견:**
(찾지 못한 파일이 있는 경우에만 표시)
- <스택트레이스 경로>: 워크스페이스에서 해당 파일을 찾지 못함

**추가 참고사항:**
<분석 과정에서 발견한 추가 참고 정보>
```

### 1-2. 보고서 작성 규칙

**지시사항:**

1. **이슈 요약 섹션**: Phase 1에서 수집한 Sentry 이슈 정보를 그대로 사용한다. 에러 타입, 발생 위치, 이벤트 수, 영향 사용자 수, Sentry 이슈 링크를 반드시 포함한다.

2. **에러 상세 섹션**: Phase 1에서 수집한 스택트레이스(inApp 프레임), 최근 이벤트 컨텍스트, 태그 정보를 포함한다.

3. **근본 원인 분석 섹션**: Phase 2에서 추정한 직접 원인, 근본 원인, 호출 체인, 관련 코드 경로를 포함한다. 관련 소스 파일 경로와 라인 번호를 반드시 명시한다.

4. **[추정] 태그**: 확실하지 않은 분석 내용에는 반드시 `[추정]` 태그를 붙여 사실과 추정을 명확히 구분한다.

5. **즉시 조치 방안 섹션**: 구체적인 코드 수정 제안 또는 설정 변경 사항을 포함한다. 단순히 "수정이 필요하다"가 아니라, 어떤 파일의 어떤 부분을 어떻게 수정해야 하는지 구체적으로 제안한다.

6. **분석 노트 섹션**: 분석 과정에서 건너뛴 단계와 그 사유, 찾지 못한 파일 등을 기록한다. 건너뛴 단계가 없으면 "없음"으로 표시한다.

## 2. 담당 파트 판단

RCA 보고서 생성 후, 코드 경로 또는 Sentry 프로젝트 정보를 기반으로 담당 프론트엔드 파트를 판단한다.

### 2-1. 파트 판단 규칙

**지시사항:**

다음 규칙에 따라 FE 파트 멘션 대상을 판단한다. 규칙은 위에서 아래로 우선순위가 높다.

1. **@creator_fe 선택 조건**: 분석 대상 코드 경로가 다음 영역에 해당하는 경우
   - 스튜디오(studio) 관련 코드 경로
   - 에디터(editor) 관련 코드 경로
   - 크리에이터 도구(creator tools) 관련 코드 경로
   - Sentry 프로젝트명에 "studio", "editor", "creator" 등이 포함된 경우

2. **@consumer_fe 선택 조건**: 분석 대상 코드 경로가 다음 영역에 해당하는 경우
   - 소비자(consumer) 관련 코드 경로
   - 뷰어(viewer) 관련 코드 경로
   - 리더(reader) 관련 코드 경로
   - Sentry 프로젝트명에 "consumer", "viewer", "reader" 등이 포함된 경우

3. **@core_fe 선택 (기본값)**: 위 조건 어디에도 해당하지 않거나, 코드 경로와 Sentry 프로젝트 정보만으로 판단할 수 없는 경우

### 2-2. 판단 결과 기록

**지시사항:**

파트 판단 결과를 다음 형식으로 기록한다:

```
👥 담당 파트 판단 결과

- 멘션 대상: <@creator_fe / @consumer_fe / @core_fe>
- 판단 근거: <어떤 코드 경로 또는 프로젝트 정보를 기반으로 판단했는지>
```

## 3. 사용자 확인 및 슬랙 게시

### 3-1. 사용자에게 보고서 및 멘션 대상 표시

**지시사항:**

1. 생성된 RCA 보고서 전체 내용을 사용자에게 표시한다.
2. 판단된 담당 파트 멘션 대상을 함께 표시한다.
3. 슬랙 스레드에 게시할지 여부를 사용자에게 확인한다.

표시 형식:

```
📄 RCA 보고서가 생성되었습니다.

(위에서 생성한 RCA 보고서 전체 내용)

---

👥 담당 파트: <멘션 대상>
판단 근거: <근거>

---

이 보고서를 원본 슬랙 스레드에 게시할까요? (멘션 포함)
- 게시하려면 "예" 또는 "게시"라고 답해주세요.
- 게시하지 않으려면 "아니오" 또는 "건너뛰기"라고 답해주세요.
```

### 3-2. 슬랙 게시 처리

**지시사항:**

1. **사용자가 승인한 경우**: Slack MCP를 통해 원본 슬랙 스레드에 RCA 보고서를 게시한다.
   - Onboarding 단계에서 파싱한 `channel_id`와 `thread_ts`를 사용한다.
   - 메시지 내용에 RCA 보고서 전체와 담당 파트 멘션을 포함한다.
   - 멘션 형식: 메시지 상단에 `<멘션 대상>` 을 포함한다.

2. **사용자가 거부한 경우**: 슬랙 게시를 건너뛰고, RCA 보고서 내용을 채팅 응답으로만 제공한다.
   - 다음 메시지를 표시한다: "슬랙 게시를 건너뛰었습니다. 위의 RCA 보고서를 참고해 주세요."

### 3-3. 에러 처리 — 슬랙 게시 실패

**지시사항:**

1. Slack MCP를 통한 메시지 게시가 실패한 경우, 다음 정보를 사용자에게 표시한다:

   ```
   ⚠️ 슬랙 스레드 게시 실패
   - 실패 사유: <에러 메시지 또는 원인>
   
   RCA 보고서 내용은 위 채팅 응답을 참고해 주세요.
   필요 시 보고서 내용을 수동으로 슬랙에 복사하여 게시할 수 있습니다.
   ```

2. 게시 실패 시에도 RCA 보고서 내용은 이미 채팅 응답으로 제공되었으므로, 사용자가 수동으로 활용할 수 있다.
