# Phase 2: 코드 분석

이 단계에서는 Phase 1에서 수집한 Sentry 스택트레이스 정보를 기반으로, Kiro IDE의 기본 파일 탐색 기능을 사용하여 워크스페이스의 소스 코드를 읽고 분석한다. 에러의 근본 원인을 추정하기 위한 코드 수준 분석 단계이다.

---

## 1. 워크스페이스 소스 파일 읽기

Phase 1에서 추출된 스택트레이스의 소스 파일 경로와 라인 번호를 기반으로, Kiro IDE의 기본 파일 탐색 기능을 통해 워크스페이스에서 해당 소스 파일을 읽는다.

### 1-1. 분석 대상 파일 우선순위

Phase 1 결과에서 추출된 파일 목록을 다음 우선순위에 따라 순서대로 읽는다:

1. **1순위 — inApp 프레임 파일**: `inApp: true`로 표시된 프레임의 파일을 먼저 읽는다. 이 파일들이 애플리케이션 코드이며, 에러의 직접적인 원인이 포함될 가능성이 가장 높다.
2. **2순위 — 스택트레이스 상단 프레임 우선**: 동일한 inApp 프레임 내에서는 스택트레이스 상단(에러 발생 지점에 가까운 프레임)의 파일을 먼저 읽는다.
3. `inApp: false`인 외부 라이브러리 프레임의 파일은 워크스페이스에서 읽지 않는다. Phase 1에서 수집한 참고 정보로만 활용한다.

### 1-2. 파일 경로 해석 및 검색

스택트레이스에 기록된 파일 경로는 빌드 환경 또는 배포 환경 기준의 경로일 수 있어, 워크스페이스의 실제 파일 경로와 다를 수 있다. 다음 절차에 따라 파일을 찾는다.

**지시사항:**

1. **직접 경로 시도**: 스택트레이스의 `filename`을 워크스페이스 상대 경로로 간주하고 `readFile` 또는 `readCode`로 직접 읽기를 시도한다.
   - 예: `filename`이 `src/components/Button.tsx`이면 워크스페이스 루트 기준 `src/components/Button.tsx`를 읽는다.

2. **경로 접두사 제거 후 재시도**: 직접 경로로 파일을 찾지 못한 경우, 스택트레이스 경로에서 흔히 포함되는 빌드/배포 접두사를 제거하고 다시 시도한다.
   - 제거 대상 접두사 예시: `app://`, `webpack://`, `webpack-internal:///`, `.next/`, `_next/`
   - 예: `webpack:///src/utils/api.ts` → `src/utils/api.ts`로 변환하여 재시도

3. **파일명 기반 검색**: 위 방법으로도 파일을 찾지 못한 경우, `fileSearch`를 사용하여 파일명으로 워크스페이스 내 검색을 수행한다.
   - 스택트레이스 경로에서 파일명 부분만 추출한다 (예: `src/components/Button.tsx` → `Button.tsx`)
   - `fileSearch`로 해당 파일명을 검색하여 매칭되는 파일을 찾는다.
   - 검색 결과가 여러 개인 경우, 스택트레이스의 디렉토리 경로와 가장 유사한 결과를 선택한다.

4. **최종 실패 처리**: 위 3단계를 모두 시도해도 파일을 찾지 못한 경우, 해당 파일에 대해 다음을 기록하고 다음 파일로 넘어간다:

   ```
   ⚠️ 해당 파일을 찾지 못함
   - 스택트레이스 경로: <filename>
   - 시도한 경로: <시도한 경로 목록>
   - 사유: 워크스페이스에 해당 파일이 존재하지 않음
   ```

### 1-3. 소스 파일 읽기 — 컨텍스트 범위

파일을 찾은 후, 에러 발생 라인뿐만 아니라 충분한 주변 컨텍스트를 함께 읽어야 한다. 단순히 해당 라인만 읽으면 에러의 맥락을 파악할 수 없다.

**지시사항:**

1. **함수/메서드 단위 읽기 (권장)**: `readCode`를 사용하여 에러 발생 라인이 포함된 함수 또는 메서드 전체를 읽는다.
   - `readCode`의 `selector` 파라미터에 스택트레이스의 `function` 이름을 전달하여 해당 함수의 전체 구현을 가져온다.
   - 예: 스택트레이스에 `function: "fetchUserData"`가 있으면 → `readCode(path="src/api/user.ts", selector="fetchUserData")`

2. **라인 범위 읽기 (대안)**: `readCode`의 selector로 함수를 찾지 못하는 경우, `readFile`로 에러 라인 주변의 충분한 컨텍스트를 읽는다.
   - 에러 발생 라인을 중심으로 **위로 30줄, 아래로 20줄** 범위를 읽는다.
   - 예: 에러 라인이 150번이면 → `readFile(path="src/api/user.ts", start_line=120, end_line=170)`

3. **파일 전체 읽기 (소규모 파일)**: 파일이 작은 경우(약 200줄 이하), `readCode`로 파일 전체를 읽어 전체 맥락을 파악한다.

4. 각 파일을 읽은 후, 다음 정보를 기록한다:
   - 읽은 파일 경로 (워크스페이스 상대 경로)
   - 읽은 라인 범위
   - 에러 발생 라인의 코드 내용
   - 해당 함수/메서드의 역할에 대한 간단한 소견

### 1-4. 분석 대상 파일 읽기 결과 정리

모든 분석 대상 파일의 읽기가 완료되면, 결과를 다음 형식으로 정리한다. 이 결과는 다음 단계(호출 체인 추적)의 입력으로 사용된다.

**결과 형식:**

```
📂 소스 파일 읽기 결과

총 분석 대상 파일: <총 파일 수>개
읽기 성공: <성공 수>개
읽기 실패 (파일 미발견): <실패 수>개

---

### 파일 #<번호>: <파일 경로>
- 스택트레이스 위치: line <lineNo>, function `<function>`
- 읽은 범위: line <start> ~ <end>
- 소견: <해당 코드의 역할 및 에러 관련 초기 소견>

(파일마다 반복)

---

(파일을 찾지 못한 경우에만 아래 섹션 표시)

### ⚠️ 파일 미발견 목록

| 스택트레이스 경로 | 시도한 경로 | 사유 |
|-------------------|-------------|------|
| <filename> | <시도한 경로들> | 워크스페이스에 해당 파일이 존재하지 않음 |
```

## 2. 호출 체인 추적 및 근본 원인 분석

섹션 1에서 읽은 소스 코드를 기반으로, 에러 발생 지점의 호출 체인을 추적하고 근본 원인을 추정한다. 단순히 에러가 발생한 라인만 보는 것이 아니라, 해당 코드가 어디서 호출되었고 어떤 데이터를 전달받았는지를 추적하여 진짜 원인을 찾는다.

### 2-1. 호출 체인 추적

스택트레이스의 프레임 순서를 따라, 에러 발생 지점에서 시작하여 상위 호출자를 추적한다.

**지시사항:**

1. **에러 발생 지점 분석 (최상단 inApp 프레임)**: 스택트레이스에서 가장 상단에 위치한 inApp 프레임의 함수를 먼저 분석한다. 이 함수가 에러를 직접 발생시킨 코드이다.
   - 해당 함수에서 에러가 발생할 수 있는 조건을 식별한다.
   - 예: null/undefined 접근, 타입 불일치, 누락된 에러 핸들링, 잘못된 API 응답 처리 등

2. **상위 호출자 추적**: 스택트레이스의 다음 프레임(호출자)으로 이동하여, 에러 발생 함수를 호출한 코드를 분석한다.
   - 호출 시 전달된 인자(arguments)가 올바른지 확인한다.
   - 호출 전에 필요한 전처리(validation, null check 등)가 수행되었는지 확인한다.
   - 스택트레이스에 나타난 모든 inApp 프레임에 대해 이 과정을 반복한다.

3. **하위 호출 대상 추적 (필요 시)**: 에러 발생 함수가 내부적으로 호출하는 다른 함수가 에러의 원인일 수 있다. 에러 발생 함수 내에서 호출하는 주요 함수를 식별하고, 필요한 경우 해당 함수의 구현도 읽는다.
   - `readCode`의 `selector`를 사용하여 하위 함수의 구현을 읽는다.
   - 하위 함수가 반환하는 값의 타입이나 에러 처리 방식을 확인한다.

4. **추가 파일 탐색**: 스택트레이스에 직접 나타나지 않았지만, 호출 체인 분석 과정에서 관련성이 높다고 판단되는 파일이 있으면 추가로 읽는다.
   - 예: import된 유틸리티 함수, 공통 API 클라이언트, 타입 정의 파일, 설정 파일 등
   - `grepSearch`를 사용하여 특정 함수나 변수의 사용처를 검색할 수 있다.

### 2-2. Next.js 프론트엔드 특화 분석

Next.js 프로젝트에서 자주 발생하는 에러 패턴에 특별히 주의하여 분석한다.

**지시사항:**

다음 항목들을 에러 원인 분석 시 우선적으로 점검한다:

1. **에러 핸들링 패턴 점검**:
   - try-catch 블록의 누락 또는 불완전한 에러 처리
   - catch 블록에서 에러를 무시(swallow)하거나 잘못 재전파하는 경우
   - Error Boundary의 부재 또는 잘못된 구성
   - API 응답의 에러 상태 코드를 처리하지 않는 경우

2. **비동기 처리 로직 점검**:
   - async/await에서 await 누락으로 인한 unhandled promise
   - Promise.all/Promise.allSettled의 부적절한 사용
   - useEffect 내 비동기 호출의 cleanup 미처리
   - 컴포넌트 언마운트 후 상태 업데이트 시도

3. **API 호출 체인 점검**:
   - fetch/axios 호출의 에러 핸들링 누락
   - API 응답 데이터의 null/undefined 체크 누락
   - Next.js API Route와 클라이언트 간 데이터 형식 불일치
   - getServerSideProps/getStaticProps에서의 에러 처리

4. **Next.js 특유 패턴 점검**:
   - 서버 컴포넌트와 클라이언트 컴포넌트 간 데이터 전달 문제
   - 하이드레이션(hydration) 불일치
   - 동적 import 실패
   - 미들웨어(middleware)에서의 에러 전파
   - next/router 또는 next/navigation 사용 시 라우팅 관련 에러

### 2-3. 근본 원인 추정

호출 체인 추적과 Next.js 특화 분석 결과를 종합하여 근본 원인을 추정한다.

**지시사항:**

1. **최소 분석 파일 수 제약**: 근본 원인 추정은 **최소 3개 이상의 관련 파일**을 분석한 후에 시작한다. 3개 미만의 파일만 분석된 경우, 추가 관련 파일을 탐색하여 분석한다.
   - 스택트레이스의 inApp 프레임 파일이 3개 미만인 경우, 호출 체인에서 관련된 추가 파일(import된 모듈, 공통 유틸리티 등)을 탐색하여 분석 대상에 포함한다.

2. **원인 추정 작성**: 분석 결과를 바탕으로 근본 원인을 추정한다. 다음 형식으로 작성한다:
   - **직접 원인**: 에러를 직접 발생시킨 코드와 조건 (예: "null 객체의 프로퍼티 접근")
   - **근본 원인**: 직접 원인이 발생하게 된 상위 원인 (예: "API 응답에서 빈 배열이 반환될 때의 처리 로직 부재")
   - **관련 코드 경로**: 원인과 관련된 파일 경로와 라인 번호 목록
   - **확신도**: 분석 결과에 대한 확신 수준 (높음 / 중간 / 낮음)
   - **추정 근거**: 해당 원인으로 추정한 이유와 근거

3. **추정 표기**: 확실하지 않은 부분은 반드시 `[추정]` 태그를 붙여 명확히 구분한다.
